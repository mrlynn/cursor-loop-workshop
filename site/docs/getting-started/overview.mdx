Cursor's product center of gravity has moved from completion to agents — most usage now runs through Agent mode. But adoption in the field lags: developers reach for one-shot prompting because it is familiar and because agentic loops feel risky and unmeasurable. Loop Engineering closes both gaps with a teachable substrate and an evidence layer.

## Why this matters

When you are the feedback loop, every iteration costs your attention. Agent mode can run commands, read test output, and correct — but only if you give it an **objective signal** to iterate against. Without a verifier, a "loop" is repeated guessing. Agents over-build poorly specified tasks, which burns trust on first contact. Loop Engineering shifts leverage from prompt-crafting to **verifier-crafting**.

## What you'll learn

- You can explain the difference between prompt mode and an agentic loop in one sentence
- You can name the four engineering surfaces: task, verifier, rules, termination
- You can identify which adoption gap (substrate vs. evidence) blocks your team today
- You can articulate the honest trade-off: loops often cost more tokens but win on pass rate and human interventions

## How it works

A traditional prompt is one-shot; the human closes the loop. An agentic loop is:

**goal → act → observe a machine-checkable signal → correct → repeat until a termination condition**

Agent mode already executes this when it runs tests and iterates on failures. The framework formalizes what makes that loop reliable: a well-scoped task, a verifier that returns pass/fail, rules that encode act→verify discipline, and termination conditions so the loop does not run forever.

The teaching payload ships on `atlas-oms` — a deliberately aged B2B Order Management System with seeded pain points. Each pain point becomes a loop task once you attach a verifier and reference solution.

## When to use this

| Situation | Prompt / Tab / Inline | Agent loop |
|-----------|----------------------|------------|
| Single-line completion | ✓ Tab | Overkill |
| Exploratory read of unfamiliar code | ✓ Chat / Plan | Optional |
| Fix with a failing test you can run | Escalate | ✓ Agent loop |
| Multi-step refactor with regression risk | Risky one-shot | ✓ Loop + full suite gate |
| Task with no machine-checkable done | ✓ Human judges | Loop will guess — craft a verifier first |

## Common mistakes

### Treating Agent mode as "Chat with terminal"
**Symptom:** Long threads, no test runs, human still judges every diff.  
**Fix:** Attach a verifier command and a termination rule before the first edit.

### Skipping the seeded failing state
**Symptom:** Agent "fixes" code that was never broken; demo falls flat.  
**Fix:** Baseline the verifier on unmodified code — it must fail first.

### Hiding loop cost
**Symptom:** Learners distrust the framework when tokens and wall-clock spike.  
**Fix:** Log cost alongside pass rate and human interventions — see Measurement section.
