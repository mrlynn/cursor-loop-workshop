The workshop ships a portable `.cursor/` directory: three loop rules and three skills. Your sample repo needs a runnable verifier before the first lab.

## Why this matters

Loop runs must be reproducible across a cohort. Same rules, same skills, same task version — otherwise comparison metrics are meaningless. Environment setup is not overhead; it is FR-6 (portable harness) in practice.

## What you'll learn

- You can run `./scripts/setup-workshop.sh` and `./scripts/copy-harness.sh` for the two-repo layout
- You can invoke skills with `/loop-engineering` and rules with `@loop-discipline`
- You can run the atlas-oms verifier locally and see a red baseline
- You can configure `.cursorignore` for secret hygiene

The workshop uses **two repositories**. This repo (`loop-engineering`) holds the curriculum site, harness, and task specs. The **atlas-oms** lab repo holds the seeded codebase where verifiers run.

## How it works

**This repo (workshop):**

```
loop-engineering/
├── .cursor/          # harness — copy into atlas-oms, do not run labs here
├── tasks/            # versioned task YAML
├── scripts/          # setup-workshop.sh, copy-harness.sh, loop-harness.sh
└── site/             # Docusaurus workshop (npm start)
```

**Lab repo (atlas-oms):** opened as the Cursor workspace for all hands-on work.

Harness contents copied into the lab:

```
.cursor/
  rules/    loop-discipline, loop-termination, loop-safety
  skills/   loop-engineering, verifier-crafting, run-comparison
```

Rules load when referenced (`@loop-discipline`). Skills load via `/skill-name`.

### One-command setup (recommended)

From the workshop repo root:

```bash
chmod +x scripts/*.sh
./scripts/setup-workshop.sh ~/code/loop-workshop
./scripts/copy-harness.sh ~/code/loop-workshop/atlas-oms
```

Then open `~/code/loop-workshop/atlas-oms` in Cursor — not the workshop repo root.

## When to use this

| Setup task | Tool | Notes |
|------------|------|-------|
| Team-wide loop discipline | Commit rules to repo | Keep always-apply lean |
| Facilitator-only comparison procedure | `run-comparison` skill | On-demand, not always-apply |
| Learner-owned repo | Copy `.cursor/` subset | Edit verifier paths per project |
| Secrets in repo | `.cursorignore` | Never index `.env` |

## Common mistakes

### Rules in personal settings only
**Symptom:** Cohort gets different agent behavior.  
**Fix:** Commit `.cursor/rules` to the repo; team syncs on pull.

### Verifier passes on seeded state
**Symptom:** Lab has no tension; loop demo fails.  
**Fix:** Wrong branch or verifier path — ask facilitator for task version pin.

### Skipping indexing
**Symptom:** Agent edits wrong files, wastes iterations.  
**Fix:** Wait for codebase indexed indicator before first loop.
